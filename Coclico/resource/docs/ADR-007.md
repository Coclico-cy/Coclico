# ADR-007 — Analyse AST via Roslyn & Clone Numérique (Digital Twin)

**Date :** 2026-03-22
**Statut :** Accepté
**Phase :** 3.3 (Ultra-Vision 105%)

---

## Contexte

La Boucle de Perfection (Phases 3.0–3.2) opère sur les **processus** (QoS CPU/RAM).
Pour atteindre l'Ultra-Vision 105% — le Méta-Gouverneur capable de s'auto-modifier —
il faut que l'IA puisse raisonner sur le **code source lui-même**, pas seulement sur
l'état d'exécution.

Sans analyse AST, l'IA est un pilote qui peut accélérer ou freiner,
mais qui ne peut pas inspecter ni réparer le moteur en vol.

---

## Décision

### 1. Roslyn comme moteur d'analyse AST

**Choix :** `Microsoft.CodeAnalysis.CSharp` (Roslyn) — le compilateur C# officiel
exposé comme bibliothèque NuGet.

**Pourquoi Roslyn et non un parser maison ?**
- API mûre, maintenue par Microsoft, 100% fidèle à la spec C#
- `SyntaxTree` immuable → thread-safe, parfait pour un clone concurrent
- Zéro dépendance réseau, zéro licence commerciale

**Métrique retenue :** Complexité Cyclomatique (McCabe, 1976)
- CC = 1 + nombre de branches (if/while/for/foreach/do/catch/switch/ternaire/&&/||)
- Seuils : Low (≤5) / Medium (6-10) / High (11-20) / Critical (≥21)
- Validé empiriquement : CC > 10 corrèle fortement avec la densité de bugs

### 2. Architecture DigitalTwin

**Pattern :** ConcurrentDictionary<filePath, (Snapshot métriques, source brute)>

**Avantage par rapport à un simple cache :**
- `SimulatePatch()` applique le remplacement textuel **en mémoire**, re-parse,
  compare ΔCC et ΔLignes — **jamais d'écriture disque**
- Garantie formelle : l'AutoPatcher ne touchera un fichier `.cs` réel que si
  `SimulatePatch().WouldImprove == true`
- Le DigitalTwin devient le **filet de sécurité mathématique** de toute modification autonome

### 3. Intégration dans l'OptimizationEngine

`ISourceAnalyzer` est injecté comme dépendance **optionnelle** dans
`OptimizationEngineService`. Le prompt LLama est enrichi avec les top-3 hotspots
du dernier rapport AST.

Résultat : LLama peut désormais croiser "CPU à 90%" avec "méthode X a CC=18" —
et décider qu'il faut throttler le processus *et* signaler la méthode pour refactoring.

---

## Conséquences

### Positives
- L'IA passe du "mode pilote" au "mode mécanicien" — elle voit le code
- La boucle est maintenant : Observer → **Analyser AST** → Agir → Valider sur Clone → Déployer
- Le Clone Numérique rend l'AutoPatcher (Phase 3.4) **safe by design**

### Négatives / Risques
- Roslyn est lourd (~15 MB de DLLs) — acceptable pour une app Desktop
- L'analyse AST d'un projet entier prend ~500ms-2s selon la taille
  → programmée en cycle asynchrone, jamais sur le thread UI
- Les snippets de méthodes pour `SimulatePatch` doivent être extraits textuellement ;
  une refactorisation qui change le nom de la méthode invalide le patch

---

## Alternatives rejetées

| Alternative | Raison du rejet |
|---|---|
| Parser regex maison | Fragile, ne supporte pas le C# moderne (records, patterns) |
| Semantic Model Roslyn complet | Requiert les références compilées — trop lourd pour la runtime |
| SonarQube / NDepend | Dépendances cloud ou commerciales, contraire au principe "zéro cloud" |
