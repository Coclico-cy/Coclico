# ADR-008 — AutoPatcher & ModuleOrchestrator (Hot-Reload)

**Date :** 2026-03-22
**Statut :** Accepté
**Phase :** 3.4–3.5 (Moteur Exécutif + Moteur Core)

---

## Contexte

Le DigitalTwin (ADR-007) peut valider mathématiquement un patch.
Il manquait les deux pièces finales du Moteur Exécutif :
1. **AutoPatcher** — qui écrit le patch validé sur disque
2. **ModuleOrchestrator** — qui recharge le module modifié en mémoire sans redémarrer

Sans ces deux services, la Boucle de Perfection ne peut qu'observer et analyser
— mais pas modifier et déployer.

---

## Décisions

### 1. AutoPatcher — Pipeline de sécurité en 3 verrous

Chaque appel à `ApplyPatchAsync` traverse 3 verrous séquentiels :

| Verrou | Mécanisme | Si échoue |
|---|---|---|
| Clone Numérique | `DigitalTwin.SimulatePatch().WouldImprove` | Rejet, log, aucune écriture |
| Snapshot Rollback | `RollbackService.CreateSnapshot()` | Annulation, log, aucune écriture |
| Écriture disque | `File.WriteAllTextAsync()` | Exception catchée, snapshot conservé pour récupération |

**Invariant absolu :** pas de `File.WriteAllTextAsync` sans les deux verrous précédents.

**Détection de drift :** avant l'écriture, on vérifie que le code source du fichier
réel correspond toujours à `originalMethodSource`. Si quelqu'un a modifié le fichier
entre l'analyse et le patch (drift), le patch est annulé avec raison "SourceDrift".

### 2. ModuleOrchestrator — AssemblyLoadContext (pas MEF)

**Choix :** `System.Runtime.Loader.AssemblyLoadContext` avec `isCollectible: true`

**Pourquoi pas MEF ?**
- `System.Composition` (MEF .NET Core) est conçu pour la découverte de plugins à l'init,
  pas pour le hot-reload en cours d'exécution
- `AssemblyLoadContext` est le primitif natif utilisé par Blazor, ASP.NET hot-reload,
  et les outils de test — c'est la solution officielle Microsoft pour .NET 6+
- Zéro NuGet supplémentaire

**HotReload atomique :**
1. Charger le nouvel assembly dans un nouveau contexte nommé
2. Mettre à jour l'index (dict) atomiquement
3. Décharger l'ancien contexte → GC peut collecter l'ancienne version

**Limitation connue :** les instances d'objets créées avec l'ancien assembly restent valides
jusqu'à leur prochain GC. Le hot-reload est "à la prochaine utilisation", pas "immédiat".
Pour Coclico, c'est acceptable car les modules sont des services Singleton récréés.

### 3. Intégration dans la Boucle de Perfection complète

```
OpenTelemetry (µs) + SAST
  → SourceAnalyzer (CC par méthode)
     → DigitalTwin (miroir AST en mémoire)
        → LLama (génère le patch textuel)
           → DigitalTwin.SimulatePatch() ← verrou 1
              → RollbackService.CreateSnapshot() ← verrou 2
                 → AutoPatcher.ApplyPatchAsync() ← écriture réelle
                    → ModuleOrchestrator.HotReload() ← déploiement
                       → Application reconfigurée en vol ✓
```

---

## Conséquences

### Positives
- La Boucle de Perfection est maintenant **complète** (Observer→Analyser→Agir→Déployer)
- Trois niveaux de sécurité avant toute modification — rollback disponible en permanence
- Le hot-reload évite tout redémarrage — continuité de service garantie

### Risques et mitigations
| Risque | Mitigation |
|---|---|
| Patch rompt la compilation | Phase 3.6 : TestRunner vérifiera dotnet build avant écriture |
| Snapshot trop volumineux | RollbackService éviction FIFO 50 entrées + pruning 30j |
| LoadContext non collecté | WeakReference monitoring (Phase 3.6) |

---

## Prochaine étape

**Phase 3.6 — TestRunner :** avant l'écriture disque, lancer `dotnet build` comme
quatrième verrou. Le patch n'est appliqué que s'il compile sans erreur.
