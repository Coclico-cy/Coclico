Architecture

Vue d'ensemble

Coclico est une application de bureau Windows construite avec .NET 10 / WPF, suivant le pattern MVVM avec injection de dépendances. L'application centralise 8 modules dans une interface unifiée Fluent Design.

Architecture

Pattern MVVM

Coclico suit le pattern Model-View-ViewModel (MVVM) via CommunityToolkit.Mvvm. Les views ne contiennent aucune logique métier — elles se limitent au data binding et à la navigation.

Views

Les views sont des UserControl WPF (XAML). Elles résolvent leurs ViewModels et services directement depuis le ServiceContainer :

// Dans DashboardView.xaml.cs
public partial class DashboardView : UserControl
{
    private readonly DashboardViewModel _vm;

    public DashboardView()
    {
        InitializeComponent();
        _vm = ServiceContainer.GetRequired<DashboardViewModel>();
        DataContext = _vm;
    }
}
C#

ViewModels

Les ViewModels héritent de ObservableObject (CommunityToolkit) et utilisent les source generators pour éviter le boilerplate :

public partial class DashboardViewModel : ObservableObject
{
    [ObservableProperty]
    private double _cpuUsage;

    [ObservableProperty]
    private string _ramUsed = "0 GB";

    [RelayCommand]
    private async Task RefreshAsync()
    {
        // Met à jour les métriques
    }
}
C#

RelayCommand vs RelayCommandAsync

Coclico définit ses propres RelayCommand et RelayCommandAsync dans Services/RelayCommand.cs et Services/RelayCommandAsync.cs. Ces wrappers implémentent ICommand avec gestion automatique du CanExecute pendant l'exécution asynchrone.

💡 Privilégiez [RelayCommand] du CommunityToolkit pour les nouvelles commandes — il génère automatiquement le code à la compilation, réduisant le boilerplate à zéro.
Architecture

DI Container

ServiceContainer est un wrapper léger autour de Microsoft.Extensions.DependencyInjection. Il expose un accès statique aux services (Instance pattern) en plus de la résolution DI classique.

Double pattern d'accès

La plupart des singletons exposent deux modes d'accès :

// Mode 1 — Via DI (recommandé, testable)
var settings = ServiceContainer.GetRequired<SettingsService>();

// Mode 2 — Accès statique direct (pratique dans les views)
var settings = SettingsService.Instance;
C#

Enregistrement des services (App.xaml.cs)

ServiceContainer.Build(services =>
{
    // Interface registration
    services.AddSingleton<ICacheService, CacheService>();

    // Singletons — partagés pour toute la durée de l'app
    services.AddSingleton<SettingsService>();
    services.AddSingleton<InstalledProgramsService>();
    services.AddSingleton<ThemeService>();
    services.AddSingleton<LocalizationService>();
    services.AddSingleton<AiChatService>();
    services.AddSingleton<KeyboardShortcutsService>();
    services.AddSingleton<FlowExecutionService>();
    services.AddSingleton<NetworkMonitorService>();
    services.AddSingleton<AppResourceGuardService>();
    services.AddSingleton<ProcessWatcherService>();
    services.AddSingleton<FeatureExecutionEngine>();
    services.AddSingleton<ProfileService>();

    // Transients — nouvelle instance à chaque résolution
    services.AddTransient<CleaningService>();
    services.AddTransient<DeepCleaningService>();
    services.AddTransient<InstallerService>();
    services.AddTransient<FlowChainService>();
    services.AddTransient<FlowChainExecutionService>();
    services.AddTransient<StartupHealthService>();
    services.AddTransient<UserAccountService>();
});

Flux de démarrage

La séquence de démarrage de Coclico est précisément orchestrée pour garantir une bonne expérience utilisateur et l'initialisation correcte de tous les services.

app.manifest
→
UAC Elevation
→
App.OnStartup()
→
DI Build
→
SplashWindow
→
LauncherWindow?
→
MainWindow

Détails de la séquence

  1. app.manifest — Demande requireAdministrator au niveau OS. Windows affiche UAC avant même que le code s'exécute.
  2. EnsureRunAsAdmin() — Vérifie programmatiquement le token Administrateur. Si absent (processus lancé sans UAC), relance avec ShellExecute runas.
  3. GC + RenderMode — Configuration des performances avant toute initialisation UI.
  4. ServiceContainer.Build() — Construction du conteneur DI. Toutes les dépendances sont enregistrées mais pas encore instanciées (lazy par défaut).
  5. SplashWindow — Affichée immédiatement. Déclenche RunStartupAsync() qui initialise les services lourds (InstalledProgramsService scan, AI model pre-check).
  6. LauncherWindow — Affichée uniquement si Settings.FirstRun == true. Permet la configuration initiale. Sauvegarde FirstRun = false à la fermeture.
  7. MainWindow — Fenêtre principale avec toute la navigation.

Modèle de données FlowChain

public class FlowChain
{
    public Guid Id { get; set; }
    public string Name { get; set; }
    public string? ShortcutKey { get; set; }       // "Ctrl+Alt+K"
    public bool UseAlt { get; set; }
    public List<FlowNode> Nodes { get; set; } = [];
    public DateTime CreatedAt { get; set; }
    public DateTime UpdatedAt { get; set; }
}

public class FlowNode
{
    public string Type { get; set; }          // "OpenApp", "Condition", etc.
    public Dictionary<string, object> Params { get; set; }
    public int RetryCount { get; set; }       // 0 = pas de retry
    public int TimeoutMs { get; set; }        // 0 = pas de timeout
    public Guid? NextNodeId { get; set; }
    public Guid? BranchTrueId { get; set; }   // Pour Condition
    public Guid? BranchFalseId { get; set; }  // Pour Condition
}

Raccourcis globaux — Implémentation

KeyboardShortcutsService installe un hook bas niveau via Win32 :

// WH_KEYBOARD_LL — hook clavier bas niveau
_hookHandle = SetWindowsHookEx(
    13, // WH_KEYBOARD_LL
    _hookCallback,
    IntPtr.Zero,
    0  // Global — tous les threads
);
C#
⚠ Le hook global clavier nécessite que le message loop Windows soit actif. Il est installé après la création de MainWindow et désinstallé proprement dans App.OnExit().

Système IA

Le système IA de Coclico est composé de 3 services qui collaborent : AiChatService, RagService, et AiActionParser.

Architecture IA

UI AiChatView.xaml AiChatViewModel
↓ Message utilisateur
RAG RagService resource/docs/*.md
↓ Contexte enrichi + prompt système
LLM AiChatService (LLamaSharp) resource/model/IA-support-chat.gguf
↓ Réponse texte + éventuelles actions
Actions AiActionParser FeatureExecutionEngine

RagService — Fonctionnement

RagService construit un index de mots-clés à partir des fichiers Markdown dans resource/docs/. Pour chaque message utilisateur :

  1. Tokenisation du message (mots-clés significatifs)
  2. Recherche dans l'index des passages les plus pertinents
  3. Les top passages sont injectés comme contexte dans le prompt système
  4. Le LLM répond avec connaissance du contexte spécifique à Coclico

Pour enrichir la base de connaissance, ajoutez des fichiers .md dans resource/docs/.

AiActionParser — Actions directes

Le modèle peut émettre des commandes d'action sous forme de tokens spéciaux dans sa réponse. AiActionParser les détecte et les route vers le FeatureExecutionEngine :

// Commandes reconnues par AiActionParser

"open_ram_cleaner"  → Navigation vers RamCleanerView
"open_scanner"      → Navigation vers ScannerView
"run_ram_clean"     → Déclenche nettoyage RAM (profil Normal)
"open_cleaning"     → Navigation vers CleaningView
"open_flowchains"   → Navigation vers FlowChainsView
"open_settings"     → Navigation vers SettingsView

MemoryCleanerService expose 18 opérations Windows natives via P/Invoke direct sur ntdll.dll, kernel32.dll et autres DLLs système.

Les 18 opérations par catégorie

foreach (var proc in Process.GetProcesses())

// Flush des pages modifiées

OpérationCode NtSetSystemInformation

// Forcer un GC complet (Gen0 + Gen1 + Gen2)

Guide Dev

Localisation

La localisation de Coclico fonctionne via deux systèmes parallèles — l'un pour l'app WPF, l'autre pour le site web.

Localisation WPF (App)

Les chaînes UI sont dans Resources/Lang/{code}.xaml — des ResourceDictionary WPF standard.

<!-- Resources/Lang/en.xaml -->
<ResourceDictionary>
    <s:String x:Key="nav_dashboard">Dashboard</s:String>
    <s:String x:Key="nav_apps">Applications</s:String>
    <s:String x:Key="btn_clean">Clean</s:String>
</ResourceDictionary>
XML

LocalizationService.SetLanguage(code) merge le dictionnaire correspondant dans Application.Current.Resources — le changement est instantané, sans redémarrage.

Ajouter une langue

  1. Dupliquez Resources/Lang/fr.xaml → xx.xaml
  2. Traduisez toutes les valeurs
  3. Ajoutez le code dans LocalizationService et le sélecteur de langue des Paramètres
  4. Créez le fichier JSON correspondant pour le site (docs/xx.json)

Localisation Site Web

Les fichiers docs/*.json contiennent les traductions pour le site. Le fichier docs/translations.js charge le JSON et remplace les éléments avec data-i18n.

Ajouter un nœud Flow Chain

Tutoriel complet pour ajouter un nouveau type de nœud — exemple : OpenBrowser.

Étape 1 — Déclarer le type dans FlowLink.cs

// Views/FlowLink.cs
public enum FlowNodeType
{
    Start, End, Condition, Loop, Delay,
    OpenApp, CloseApp, KillProcess,
    // ... autres types existants ...
    OpenBrowser  // ← Nouveau type
}

public static class FlowNodeMeta
{
    public static FlowNodeDefinition Get(FlowNodeType type) => type switch
    {
        // ... existants ...
        FlowNodeType.OpenBrowser => new("OpenBrowser", "Ouvrir un URL", NodeColor.Green,
            new[] { "Url" }),  // Paramètres requis
        _ => throw new ArgumentOutOfRangeException()
    };
}
C#

Étape 2 — Implémenter dans FlowChainExecutionService

// Services/FlowChainExecutionService.cs
private async Task<bool> ExecuteNodeAsync(FlowNode node, CancellationToken ct)
{
    return node.Type switch
    {
        // ... cas existants ...
        "OpenBrowser" => await ExecuteOpenBrowserAsync(node, ct),
        _ => throw new InvalidOperationException($"Unknown node: {node.Type}")
    };
}

private async Task<bool> ExecuteOpenBrowserAsync(FlowNode node, CancellationToken ct)
{
    var url = node.Params.GetValueOrDefault("Url")?.ToString();
    if (string.IsNullOrEmpty(url)) return false;
    Process.Start(new ProcessStartInfo(url) { UseShellExecute = true });
    return true;
}
C#

Ajouter un Service

Processus recommandé pour créer un nouveau service dans l'architecture de Coclico.

1. Créer le service

// Services/MyNewService.cs
namespace Coclico.Services;

public class MyNewService
{
    public static MyNewService? Instance { get; private set; }

    public MyNewService(SettingsService settings)
    {
        Instance = this;
        // Injection des dépendances via constructeur
    }

    public async Task<string> DoSomethingAsync()
    {
        // Logique métier
        return await Task.FromResult("result");
    }
}
C#

2. Enregistrer dans App.xaml.cs

// Singleton si état partagé :
services.AddSingleton<MyNewService>();

// Transient si opération courte :
services.AddTransient<MyNewService>();
C#

3. Utiliser le service

// Dans un ViewModel ou une View :
var svc = ServiceContainer.GetRequired<MyNewService>();
var result = await svc.DoSomethingAsync();

// Ou via le pattern Instance :
var result = await MyNewService.Instance!.DoSomethingAsync();
